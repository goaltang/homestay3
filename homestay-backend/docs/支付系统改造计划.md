# 民宿系统支付宝微信沙箱支付改造计划

## 1. 现状分析

### 1.1 当前支付系统特点

- **现有实现**: 基于模拟支付，使用第三方二维码生成服务生成假的支付二维码
- **支付状态**: 支持 UNPAID, PAID, PAYMENT_FAILED 等状态管理
- **支付流程**: 订单创建 → 生成二维码 → 模拟支付 → 状态更新
- **技术特点**: PaymentService 接口抽象良好，便于扩展
- **数据结构**: Order 实体已包含 paymentStatus 和 paymentMethod 字段

### 1.2 系统优势

✅ **架构良好**: PaymentService 接口设计合理，方便替换实现  
✅ **状态完整**: PaymentStatus 枚举支持完整的支付生命周期  
✅ **业务完善**: 已有完整的订单状态转换逻辑  
✅ **前端支持**: 支付页面和状态检查逻辑已实现  
✅ **权限安全**: 支付接口已有合理的权限控制

### 1.3 需要改造的部分

❌ **支付网关**: 当前为模拟支付，需要接入真实支付宝微信 API  
❌ **异步回调**: 缺少支付结果异步通知处理  
❌ **配置管理**: 缺少支付宝微信的配置参数  
❌ **安全签名**: 缺少支付安全验证机制  
❌ **退款处理**: 当前退款为状态变更，需要对接真实退款 API

## 3. 技术方案设计

### 3.1 整体架构

```
前端支付页面 → Spring Boot后端 → 支付网关适配器 → 支付宝/微信沙箱API
                                  ↓
                          异步回调处理 → 订单状态更新
```

### 3.2 核心组件设计

#### 3.2.1 支付网关抽象

```java
public interface PaymentGateway {
    PaymentResponse createPayment(PaymentRequest request);
    PaymentStatusResponse queryPaymentStatus(String outTradeNo);
    RefundResponse refund(RefundRequest request);
    boolean verifyCallback(Map<String, String> params);
}
```

#### 3.2.2 支付配置

```properties
# 支付宝沙箱配置
alipay.sandbox.app-id=沙箱APPID
alipay.sandbox.private-key=商户私钥
alipay.sandbox.public-key=支付宝公钥
alipay.sandbox.gateway-url=https://openapi.alipaydev.com/gateway.do

# 微信沙箱配置
wechat.sandbox.app-id=沙箱AppId
wechat.sandbox.mch-id=商户号
wechat.sandbox.api-key=API密钥
wechat.sandbox.cert-path=证书路径
```

### 3.3 数据库扩展

需要新增支付流水表记录详细的支付信息：

```sql
CREATE TABLE payment_records (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    order_id BIGINT NOT NULL,
    payment_method VARCHAR(20) NOT NULL,
    out_trade_no VARCHAR(64) NOT NULL UNIQUE,
    transaction_id VARCHAR(64),
    amount DECIMAL(10,2) NOT NULL,
    status VARCHAR(20) NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);
```

## 9. 结论

基于当前系统分析，将民宿系统改造为支持支付宝微信沙箱支付的难度为**中等偏易**。主要原因：

### 9.1 有利因素

- 现有架构设计良好，支付服务接口抽象合理
- 订单状态管理完善，支持复杂的支付流程
- 前端支付页面已经完备，改动量小
- 团队有 Spring Boot 开发经验，技术栈匹配

### 9.2 主要挑战

- 需要处理真实支付网关的复杂性
- 异步回调处理需要考虑各种边界情况
- 安全性要求更高，需要严格的签名验证

### 9.3 总体评价

这是一个**高价值、中等风险、可控难度**的改造项目，预计**6-8 天**可以完成，建议按计划推进。改造完成后，系统将具备真实的支付能力，为业务发展奠定坚实基础。
