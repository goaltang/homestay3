# 🏡 推荐民宿和热门民宿功能完整改进计划

## 📊 项目现状分析

### 现有实现问题

1. **算法简陋**：推荐和热门民宿都使用相同的 `featured=true` 标识，缺乏差异化
2. **无缓存机制**：每次请求都直接查询数据库，性能较差
3. **无个性化推荐**：没有基于用户行为的智能推荐
4. **前后端耦合度高**：前端组件直接调用通用 API，没有专门的推荐接口

### 技术栈现状

- **后端**：Spring Boot 3.0.2 + JPA + MySQL
- **前端**：Vue 3 + TypeScript + Element Plus
- **缓存**：新增 Redis 支持

## 🔧 技术实现细节

### 推荐算法权重配置

```java
// 热门度计算权重
WEIGHT_BOOKING_COUNT = 0.4    // 预订量权重
WEIGHT_RATING = 0.3           // 评分权重
WEIGHT_REVIEW_COUNT = 0.2     // 评论数权重
WEIGHT_RECENCY = 0.1          // 时间衰减权重

// 推荐度计算权重
质量评分权重 = 0.5
活跃度评分权重 = 0.3
性价比评分权重 = 0.2
```

### 缓存键命名规范

```
popularHomestays:{limit}
recommendedHomestays:{limit}
userRecommendations:{userId}_{limit}
searchResults:{hash}
homestayDetails:{id}
```

### 个性化推荐流程

1. **用户行为分析**：分析历史订单、浏览记录、收藏等
2. **偏好建模**：提取价格区间、位置、房型、设施偏好
3. **候选集筛选**：基于偏好筛选候选民宿
4. **评分计算**：结合基础推荐分数和个性化加权
5. **结果排序**：按最终分数排序并返回

### 性能优化策略

1. **数据库层**：

   - 添加复合索引：`(status, created_at)`, `(province_code, city_code, status)`
   - 使用连接查询减少 N+1 问题
   - 实现分页查询优化

2. **应用层**：

   - 使用 `@Async` 异步计算推荐
   - 实现批量数据加载
   - 添加限流和降级机制

3. **缓存层**：
   - 多级缓存：本地缓存 + Redis 分布式缓存
   - 缓存预热：定时任务预计算热门推荐
   - 缓存更新：增量更新机制
