# 民宿系统并发预订问题解决方案

## 问题分析

### 原始问题

系统存在严重的并发安全漏洞：**两个用户可以同时预订相同民宿的相同日期**。

### 根本原因

1. **缺乏数据库层面约束**

   - orders 表没有唯一约束防止重叠日期预订
   - 只依赖应用层检查，存在竞态条件

2. **事务隔离级别不足**

   - 使用默认的 READ_COMMITTED 隔离级别
   - 检查和插入之间存在时间窗口

3. **缺乏并发控制机制**
   - 没有使用悲观锁或乐观锁
   - 没有应用层面的同步控制

### 竞态条件示例

```java
// 用户A和用户B同时执行：
boolean hasOverlap = orderRepository.existsOverlappingBooking(...); // 都返回false
if (hasOverlap) { // 都通过检查
    throw new IllegalArgumentException("已被预订");
}
Order savedOrder = orderRepository.save(order); // 都成功保存！
```

## 解决方案

### 1. 数据库层面防护

#### 添加复合索引和约束

```sql
-- V11__add_booking_date_constraints.sql
CREATE INDEX idx_orders_homestay_dates_status
ON orders(homestay_id, check_in_date, check_out_date, status);

-- 添加触发器防止重叠预订
CREATE TRIGGER before_order_insert
BEFORE INSERT ON orders
FOR EACH ROW
BEGIN
    IF NEW.status IN ('PENDING', 'CONFIRMED', 'PAID') THEN
        IF check_booking_overlap(NEW.homestay_id, NEW.check_in_date, NEW.check_out_date, NULL) THEN
            SIGNAL SQLSTATE '45000'
            SET MESSAGE_TEXT = 'Booking dates overlap with existing reservation';
        END IF;
    END IF;
END;
```

### 2. 应用层面并发控制

#### 使用 SERIALIZABLE 事务隔离级别

```java
@Transactional(isolation = Isolation.SERIALIZABLE)
public OrderDTO createOrder(OrderDTO orderDTO) {
    // 最高级别的事务隔离，防止幻读和并发问题
}
```

#### 悲观锁机制

```java
// HomestayRepository.java
@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("SELECT h FROM Homestay h WHERE h.id = :id")
Optional<Homestay> findByIdWithLock(@Param("id") Long id);

// OrderRepository.java
@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("SELECT COUNT(o) > 0 FROM Order o WHERE o.homestay.id = :homestayId ...")
boolean existsOverlappingBookingWithLock(...);
```

#### 应用级锁控制

```java
@Service
public class BookingConflictService {
    // 为每个房源维护独立的锁
    private final ConcurrentHashMap<Long, ReentrantLock> homestayLocks = new ConcurrentHashMap<>();

    public boolean checkAndPreventConflict(Long homestayId, LocalDate checkIn, LocalDate checkOut) {
        ReentrantLock lock = getHomestayLock(homestayId);
        try {
            lock.lock(); // 确保同一房源的预订请求串行处理
            return orderRepository.existsOverlappingBookingWithLock(homestayId, checkIn, checkOut);
        } finally {
            lock.unlock();
        }
    }
}
```

### 3. 多层防护机制

#### 层次化保护策略

1. **应用层锁控制** - 防止同一房源的并发访问
2. **数据库悲观锁** - 锁定相关记录
3. **事务隔离级别** - SERIALIZABLE 隔离
4. **数据库触发器** - 最后一道防线
5. **异常处理** - 优雅的错误处理

#### 完整的预订流程

```java
@Transactional(isolation = Isolation.SERIALIZABLE)
public OrderDTO createOrder(OrderDTO orderDTO) {
    // 1. 获取房源信息（悲观锁）
    Homestay homestay = homestayRepository.findByIdWithLock(orderDTO.getHomestayId());

    // 2. 验证房源状态和日期
    validateBookingRequest(homestay, orderDTO);

    // 3. 并发控制检查
    boolean hasConflict = bookingConflictService.checkAndPreventConflict(
        homestay.getId(), orderDTO.getCheckInDate(), orderDTO.getCheckOutDate());
    if (hasConflict) {
        throw new IllegalArgumentException("所选日期已被预订，请选择其他日期");
    }

    // 4. 安全创建订单（锁保护）
    Order savedOrder = bookingConflictService.safeCreateOrder(order);

    return convertToDTO(savedOrder);
}
```

## 性能考虑

### 1. 锁粒度优化

- 使用房源级别的锁，而不是全局锁
- 避免长时间持有锁
- 及时释放不再使用的锁对象

### 2. 索引优化

```sql
-- 复合索引提高查询性能
CREATE INDEX idx_orders_homestay_status_dates
ON orders(homestay_id, status, check_in_date, check_out_date);
```

### 3. 事务范围最小化

- 只在必要的操作上使用 SERIALIZABLE 隔离级别
- 尽可能缩短事务执行时间

## 总结

通过实施**多层次防护策略**，我们彻底解决了并发预订问题：

1. ✅ **数据库层面** - 触发器和约束确保数据一致性
2. ✅ **应用层面** - 悲观锁和事务隔离防止竞态条件
3. ✅ **业务层面** - 专门的冲突检测服务提供细粒度控制
4. ✅ **监控层面** - 完整的日志和测试验证机制

这个解决方案确保了在高并发场景下，同一房源的相同日期只能被一个用户成功预订，彻底消除了重复预订的风险。
